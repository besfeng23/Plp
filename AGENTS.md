# AGENTS.md

Use this file for Codex and other coding agents working on the Pueblo La Perla Boracay repo.

## Project identity

Pueblo La Perla Boracay is a private hillside luxury resort website with a public guest-facing site and an internal Resort Command admin system.

The public site should feel calm, editorial, credible, mobile-first, and conversion-aware. The admin system should feel operational, records-first, secure, and useful for real reservation desk work.

## Public-site rules

For public guest-facing work involving UX, navigation, copy, images, routes, gallery, booking entry, SEO, sitemap, or mobile layout:

- Do not turn the homepage into a content dump.
- Every page needs a clear job.
- Every section needs a reason.
- Every CTA needs correct timing.
- Do not use generic luxury filler copy.
- Do not invent property facts, phone numbers, reviews, awards, amenities, addresses, transfer guarantees, response times, exact distances, or exact travel times.
- Do not add image or video paths unless the file exists or is generated by a repo script.
- Do not claim a route, nav item, gallery, or image is complete until the affected React and static layers have been checked.

## Admin-system rules

Admin work lives primarily around the Resort Command surfaces.

Known admin files may include:

- `src/admin/OpsAdminApp.jsx`
- `public/admin-ops.html`
- `/api/admin` routes and related server files

Admin priorities:

1. Reservations and Reservation 360
2. Today command dashboard
3. Payment review clarity
4. Guest and arrival notes
5. Availability and housekeeping
6. Concierge tasks
7. Content controls
8. Staff and role controls

When building admin features:

- Preserve records-first behavior.
- Preserve raw record fallbacks.
- Clearly separate real persisted data from UI-only placeholders.
- Do not imply internal notes, staff assignments, tasks, or guest memory are saved unless persistence exists.
- Use existing loaded fields first before inventing new schema requirements.
- Avoid large rewrites. Prefer small, reviewable PRs.

## Booking and payment danger zones

Treat these as high risk:

- Public booking form behavior
- Booking creation
- Booking status semantics
- Payment status semantics
- Deposit verification
- Xendit checkout
- Xendit webhooks
- Supabase service-role logic
- Admin booking-status mutation
- Any reconciliation or payment exception logic

For normal UX/admin UI work:

- Do not touch payment/webhook/Supabase service-role logic.
- Do not change payment verification meaning.
- Do not change confirmed/cancelled/fully paid semantics.
- Do not imply instant confirmation if the process is review-first.
- Do not remove audit or raw-record visibility.

If a task explicitly requires touching these areas, isolate the change, explain the risk, and add stronger validation.

## Static vs React consistency

PLP has React-rendered views and static HTML route files. When changing navigation, contact/concierge, booking entry, footer links, image placement, policies, sitemap routes, or public emails, check the relevant layers:

- React route map and views
- React desktop nav
- React mobile menu
- React footer
- React mobile action bar
- Static HTML navs
- Static mobile quick-nav/action bars
- `public/content-hydrate.js`
- `public/sitemap.xml`
- `vercel.json`
- Route audit script behavior

If only one layer was changed, say so. Never overclaim.

## Asset verification

Before referencing any public asset, prove at least one of these is true:

- The file exists under `public/images` or `public/videos` and is tracked.
- The file is generated by `scripts/optimize-plp-images.mjs` from a tracked source.
- The file is already referenced successfully by existing production code.

A passing build does not prove public asset paths work.

## Required validation

Before claiming readiness, run when available:

```bash
npm run build
npm run check:viewport
npm run check:routes
```

Interpret results honestly.

Known route-audit nuance:

- `/privacy` may intentionally redirect to `/privacy-policy`.
- `/terms` may intentionally redirect to `/terms-of-service`.

If those appear as route-audit failures, report them as known redirect-audit issues, not new regressions, unless the redirect behavior itself changed.

## PR rules

Every PR should be small and honest.

PR body should include:

- What changed
- Files changed
- Whether public-site behavior changed
- Whether admin behavior changed
- Whether booking/payment/Supabase/Xendit/webhook logic was touched
- What validation was run
- Known remaining risks

Do not say “all checks pass” unless all checks truly pass.

Do not say “production verified” unless live production was checked.

Do not merge or recommend merge if:

- PR claims are false or unverified
- Image paths are unproven
- React/static route drift is likely
- Payment/booking logic was touched out of scope
- Fake facts were introduced
- Mobile layout risk was ignored
- Build/route failures are hidden or softened
- The PR mixes unrelated concerns

## Branching rule

Start new work from `main` unless explicitly continuing an open PR branch.

Do not base new work on stale PR branches.
