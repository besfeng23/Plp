# Claude Code Project Guardrails

Use this file when working in Claude Code on Pueblo La Perla Boracay.

## Project identity

Pueblo La Perla Boracay is a private hillside luxury resort website. The public site should feel calm, editorial, credible, mobile-first, and conversion-aware. Do not turn the homepage into a dumping ground.

## Required skills

For public-site UX, content architecture, resort design, gallery, navigation, booking-page copy, image placement, route changes, sitemap changes, or luxury hospitality decisions, use:

`world-class-luxury-resort-ui-ux`

The project skill lives at:

`.claude/skills/world-class-luxury-resort-ui-ux/SKILL.md`

For PR review, merge readiness, deployment checks, production smoke tests, route verification, static/dynamic drift review, asset verification, booking/payment safety checks, or PR claim accuracy, use:

`plp-production-reviewer`

The production reviewer skill lives at:

`.claude/skills/plp-production-reviewer/SKILL.md`

## Non-negotiables

- Every page needs a clear job.
- Every section needs a reason.
- Every CTA needs correct timing.
- Every homepage item must earn its place.
- Do not add fake phone numbers, fake reviews, fake awards, fake amenities, fake maps, fake addresses, fake distances, fake response times, or fake policies.
- Do not invent property facts.
- Do not overload the homepage to solve weak information architecture.
- Do not use generic luxury filler copy.
- Do not touch payment, webhook, booking status, Supabase service-role, or Xendit verification logic unless the task explicitly requests backend payment work.
- Do not claim a navigation, route, or gallery change is complete until both React and static production routes have been checked.
- Do not reference an image/video/file path unless it exists in the repo or is generated by a build script.
- Do not treat a passing build as proof that public asset paths are valid.
- Do not approve or recommend merging a PR until the PR body, route behavior, static/dynamic consistency, generated assets, and validation results have been checked.

## Static/dynamic consistency rule

PLP has both React-rendered views and static HTML route files. When changing navigation, gallery, contact/concierge, booking entry, footer links, policies, image placement, or sitemap routes, verify every affected layer:

- React route map and view selection
- React desktop nav
- React mobile menu
- React footer
- Static HTML navs
- Static mobile quick-nav/action bars
- Vercel rewrites
- Sitemap/canonical URLs
- Route audit script, when relevant

If only one layer was changed, say so. Never overclaim.

## Asset verification rule

Before adding any image or video reference, confirm at least one of these is true:

- The file exists under `public/images` or `public/videos` and is tracked.
- The file is generated by `scripts/optimize-plp-images.mjs` from a tracked source.
- The file is already referenced successfully by existing production code.

Broken public asset paths can still pass `npm run build`. Check them manually.

## Required validation

Run these before claiming readiness:

```bash
npm run build
npm run check:viewport
npm run check:routes
```

If `npm run check:routes` fails because the execution environment cannot reach Vercel, say that clearly and do not pretend live production was verified.

If `npm run check:routes` audits live production while the current branch is not deployed, say that clearly. If failures are known intentional redirects, name the exact routes.

## PR summary rule

Before writing or accepting a PR summary, verify the summary against the diff. If a claim is only partially true, write it as partially true. Example:

Bad:

`Gallery is wired into navigation.`

Good:

`Gallery is wired into React nav/footer; static accommodation and experiences nav still need follow-up.`

## Branching rule

Start new luxury UX work from `main`, not from stale PR branches. Existing old PRs may touch the same files and should not be used as the base unless they are intentionally revived and rebased.
